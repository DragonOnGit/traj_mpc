# Traj MPC Package

基于ROS1的MPC轨迹跟踪控制功能包（无MAVROS依赖），使用QP优化求解器实现模型预测控制，采用一阶质点动力学模型。

## 功能特点

1. **MPC参数可调节**：通过launch文件或ROS参数服务器调整MPC控制器参数
2. **三维独立参数**：X/Y/Z轴方向的控制参数可单独配置，实现精细化控制
3. **QP优化求解**：集成二次规划(QP)优化方法作为MPC核心求解器
4. **一阶质点动力学模型**：基于牛顿第二定律的双积分器模型，直接以加速度为控制输入
5. **速度控制**：MPC计算最优加速度，转换为速度指令通过/cmd_vel话题发布
6. **重力补偿**：可配置的Z轴重力前馈补偿，适配不同底层控制器
7. **单一指令发布**：每个控制周期仅发布一条速度指令，避免指令冲突
8. **XML轨迹定义**：期望轨迹由XML文件定义，关键点使用point标签表示
9. **四元数姿态**：轨迹点包含位置和姿态四元数信息
10. **无MAVROS依赖**：适用于各种机器人平台，不依赖特定的飞控系统
11. **参考轨迹发布**：将参考轨迹数据发布至/path_exp话题，便于可视化和调试
12. **期望位置发布**：将期望轨迹当前位置发布至/position_exp话题

## 模型变更说明（v2.0）

### 旧模型：二阶弹簧-阻尼系统

```
x'' + 2·ζ·ω·x' + ω²·x = ω²·u
```

旧模型存在的问题：
- 控制输入u通过ω²缩放，导致有效增益随频率参数变化
- 弹簧-阻尼模型不适用于飞行器动力学，无法准确反映质点运动
- 阻尼比ζ和自然频率ω的物理含义与飞行器不匹配
- 状态空间矩阵A、B构建后未在MPC求解中实际使用，模型与优化不一致

### 新模型：一阶质点动力学模型（双积分器）

```
p' = v
v' = a
```

其中：
- `p`：位置
- `v`：速度
- `a`：加速度（控制输入 = 合力/质量）

离散化后的状态空间方程：

```
[p(k+1)]   [1  dt ] [p(k)]   [0.5·dt²] [a(k)]
[v(k+1)] = [0  1  ] [v(k)] + [dt      ] [a(k)]
```

新模型优势：
- 直接以加速度为控制输入，物理含义清晰
- 模型参数与实际物理特性完全匹配
- 状态空间模型在MPC求解中完整使用，预测与优化一致
- 可通过重力前馈补偿适配不同底层控制器

### 参数变更对照

| 旧参数 | 新参数 | 说明 |
|--------|--------|------|
| damping_x/y/z | （已移除） | 弹簧-阻尼模型参数不再需要 |
| omega_x/y/z | （已移除） | 弹簧-阻尼模型参数不再需要 |
| max_acc_x/y/z (默认1.0) | max_acc_x/y/z (默认3.0) | 加速度约束放宽以适配质点模型 |
| （无） | gravity (默认9.81) | 重力加速度参数 |
| （无） | need_gravity_compensation (默认false) | 重力前馈补偿开关 |

## MPC控制算法

### 一阶质点模型

系统采用一阶质点动力学模型（双积分器）：

```
m·a = F  →  a = F/m
```

状态向量：`x = [px, py, pz, vx, vy, vz]`（6维）
控制输入：`u = [ax, ay, az]`（3维）

离散状态空间方程（零阶保持）：

```
A_axis = [1  dt]    B_axis = [0.5·dt²]
         [0  1 ]             [dt      ]

A_full = blkdiag(A_axis, A_axis, A_axis)  (6×6)
B_full = blkdiag(B_axis, B_axis, B_axis)  (6×3)
```

### 目标函数

MPC优化的目标函数为：

```
J = Σ_{k=1}^{N} [ (p(k) - p_ref(k))^T · P · (p(k) - p_ref(k))
                 + v(k)^T · Q · v(k) ] + Σ_{k=0}^{N-1} a(k)^T · R · a(k)
```

其中：
- `p(k)`：预测位置（3维）
- `v(k)`：预测速度（3维）
- `a(k)`：控制加速度（3维）
- `P`：位置误差权重矩阵（3×3对角阵）
- `Q`：速度权重矩阵（3×3对角阵）
- `R`：加速度权重矩阵（3×3对角阵）

### 状态空间提升与QP求解

使用状态空间提升（lifting）技术，将N步预测表示为：

```
X_pred = F · x₀ + G · U
```

其中：
- `X_pred = [x(1); x(2); ...; x(N)]`（6N维预测状态向量）
- `U = [a(0); a(1); ...; a(N-1)]`（3N维控制输入向量）
- `F`：状态传播矩阵（6N×6）
- `G`：输入传播矩阵（6N×3N）

F和G矩阵通过预计算A的幂次得到：

```
F(k) = A^{k+1}
G(k, i) = A^{k-i} · B  (i ≤ k)
```

将目标函数转化为标准QP形式：

```
J = U^T · (G^T·Q̄·G + R̄) · U + 2·(F·x₀ - X_ref)^T·Q̄·G · U + const
```

标准QP：`min 0.5·U^T·H·U + f^T·U`

```
H = 2·(G^T·Q̄·G + R̄)
f = 2·G^T·Q̄·(F·x₀ - X_ref)
```

约束条件：
- 加速度约束：`-max_acc ≤ a ≤ max_acc`（各轴独立）
- 速度约束：通过收紧第一步加速度边界实现

### 速度指令生成

MPC求解得到最优加速度后，转换为速度指令：

```
v_cmd = v_current + a_cmd · dt
```

Z轴重力前馈补偿（可选）：

```
v_cmd_z = v_current_z + a_cmd_z · dt + g · dt  （当need_gravity_compensation=true时）
```

## 弹跳问题修复说明

### 问题根因

设备在地面持续弹跳的根本原因是**双重控制指令发布**：

1. **P控制器**：在`controlCallback`中计算 `v = Kp * e` 并发布到`/cmd_vel`
2. **MPC控制器**：同时通过`computeControl()`发布另一条速度指令到`/cmd_vel`

每个控制周期内，设备收到两条不同的速度指令，导致指令冲突和振荡。

### 修复措施

1. **移除P控制器**：删除`controlCallback`中的简单比例控制代码
2. **单一指令发布**：每个控制周期仅由MPC计算并发布一条速度指令
3. **模型重构**：从二阶弹簧-阻尼模型改为一阶质点模型，确保预测与控制一致
4. **重力补偿**：添加可配置的Z轴重力前馈，防止设备因重力下落
5. **加速度约束调整**：默认最大加速度从1.0 m/s²提高到3.0 m/s²，确保有足够动力起飞

## MPC参数配置

### 基本参数

| 参数 | 默认值 | 描述 |
|------|--------|------|
| horizon | 10 | MPC预测步长 |
| dt | 0.1 | 时间步长（秒） |
| control_rate | 10.0 | 控制频率（Hz） |

### 三维位置误差权重（P矩阵对角元素）

| 参数 | 默认值 | 描述 |
|------|--------|------|
| weight_pos_x | 10.0 | X轴位置误差权重 |
| weight_pos_y | 10.0 | Y轴位置误差权重 |
| weight_pos_z | 10.0 | Z轴位置误差权重 |

**调节建议**：
- 较大的值：优先保证该轴的位置跟踪精度
- 较小的值：允许较大的位置误差，系统更稳定
- 不同轴可设置不同值以适应不同方向的精度需求

### 三维速度权重（Q矩阵对角元素）

| 参数 | 默认值 | 描述 |
|------|--------|------|
| weight_vel_x | 1.0 | X轴速度权重 |
| weight_vel_y | 1.0 | Y轴速度权重 |
| weight_vel_z | 1.0 | Z轴速度权重 |

**调节建议**：
- 较大的值：速度变化更平滑，减少抖动
- 较小的值：速度响应更快速，跟踪更灵敏

### 三维加速度权重（R矩阵对角元素）

| 参数 | 默认值 | 描述 |
|------|--------|------|
| weight_acc_x | 0.1 | X轴加速度权重 |
| weight_acc_y | 0.1 | Y轴加速度权重 |
| weight_acc_z | 0.1 | Z轴加速度权重 |

**调节建议**：
- 较大的值：加速度变化更平滑，控制更温和
- 较小的值：加速度响应更快速，但可能引起振荡

### 三维速度约束

| 参数 | 默认值 | 描述 |
|------|--------|------|
| max_vel_x | 2.0 | X轴最大速度 (m/s) |
| max_vel_y | 2.0 | Y轴最大速度 (m/s) |
| max_vel_z | 2.0 | Z轴最大速度 (m/s) |

### 三维加速度约束

| 参数 | 默认值 | 描述 |
|------|--------|------|
| max_acc_x | 3.0 | X轴最大加速度 (m/s²) |
| max_acc_y | 3.0 | Y轴最大加速度 (m/s²) |
| max_acc_z | 3.0 | Z轴最大加速度 (m/s²) |

**调节建议**：
- 需确保最大加速度大于重力加速度(9.81 m/s²)才能实现垂直起飞
- 较大的值：机动性更强，但可能引起机械应力
- 较小的值：运动更温和，但起飞和机动能力受限

### 重力参数

| 参数 | 默认值 | 描述 |
|------|--------|------|
| gravity | 9.81 | 重力加速度 (m/s²) |
| need_gravity_compensation | false | 是否需要Z轴重力前馈补偿 |

**need_gravity_compensation调节建议**：
- `false`（默认）：底层速度控制器已包含重力补偿（如PX4、ArduPilot的定高模式）
- `true`：底层速度控制器不包含重力补偿，MPC在Z轴速度指令中添加 `g·dt` 前馈
- 如果设备在悬停时持续下落，尝试设置为`true`

## 参考轨迹发布

- **话题名称**: `/path_exp`
- **消息类型**: `nav_msgs/Path`
- **发布频率**: 10Hz
- **数据格式**: 包含轨迹中所有路标点的PoseStamped消息数组
- **坐标系**: camera_init

## 期望位置发布

- **话题名称**: `/position_exp`
- **消息类型**: `nav_msgs/Odometry`
- **发布频率**: 与控制频率一致（默认10Hz）
- **数据格式**: 当前期望轨迹点的位置和姿态信息
- **坐标系**: camera_init
- **发布逻辑**:
  - 初始化时发布首个路标点
  - 路标点更新时立即发布新点
  - 未更新时持续发布当前点

## 控制模式

系统使用**MPC-only速度控制模式**进行轨迹跟踪：

1. **MPC计算**：使用一阶质点模型，通过状态空间提升和QP求解器计算最优加速度序列
2. **加速度转速度**：`v_cmd = v_current + a_cmd · dt`
3. **重力补偿**：可选的Z轴重力前馈（`need_gravity_compensation`参数控制）
4. **安全限幅**：速度指令被限制在`max_vel`范围内
5. **单一发布**：每个控制周期仅发布一条速度指令到`/cmd_vel`

```bash
roslaunch traj_mpc traj_mpc.launch
```

## 程序执行流程

1. **初始化阶段**：
   - 加载MPC参数（三维独立参数）和话题配置
   - 构建一阶质点模型的状态空间矩阵A、B
   - 预计算预测矩阵F、G
   - 构建权重矩阵P、Q、R
   - 加载XML轨迹文件
   - 初始化订阅器和发布器
   - 进入IDLE状态，等待里程计数据

2. **轨迹跟踪阶段**：
   - 接收到第一条里程计消息后，自动开始轨迹跟踪
   - 基于当前航路点生成参考轨迹
   - 构建全时域QP问题：计算H矩阵和f向量
   - 通过QP求解器求解最优加速度序列
   - 取第一步加速度转换为速度指令
   - 应用重力补偿（可选）和安全限幅
   - 发布单一速度控制指令

3. **任务完成阶段**：
   - 所有路标点访问完成后，进入COMPLETED状态
   - 发布零速度指令停止运动

## 轨迹文件格式

轨迹文件使用XML格式，示例如下：

```xml

<trajectory>
  <point x="0.0" y="0.0" z="1.0" qx="0.0" qy="0.0" qz="0.0" qw="1.0" />
  <point x="1.0" y="0.0" z="1.0" qx="0.0" qy="0.0" qz="0.0" qw="1.0" />
</trajectory>
```
